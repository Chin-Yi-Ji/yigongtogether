## Rate limiting 說明（深入）

### 什麼是 Rate limiting？

**Rate limiting（速率限制）** 是指：在一個時間區間內，**同一個來源**（同一 IP、同一使用者、同一 API Key 等）對某個 API **最多只能請求 N 次**；超過就拒絕並回傳 429 Too Many Requests，直到時間過了或計數重置。

- **目的**：避免單一來源短時間內瘋狂打 API，造成伺服器負載、資料庫壓力，或被用來暴力嘗試、爬資料、濫用註冊／上傳等。
- **「來源」** 常見做法：依 **IP**（未登入）、或 **userId / session**（已登入）。YigongHub 可先以 IP 為主，登入後可再加「同一使用者」限制。

### YigongHub 建議限制的對象

| API／行為 | 為什麼要限流 | 建議方向（僅供參考） |
|-----------|--------------|------------------------|
| **登入／OAuth** | 防止暴力嘗試、惡意佔用登入頁 | 同一 IP 每分鐘最多 N 次（例如 10～20 次） |
| **建立 Profile** `POST /api/profile/create` | 一人只需建立一次，重複送可能是爬蟲或異常 | 同一 IP 或同一 userId 每小時／每天 上限 |
| **上傳大頭照** `POST /api/upload/avatar` | 避免有人狂傳大檔案把頻寬或 R2 打滿 | 同一使用者每分鐘／每小時 最多幾次（例如 5～10 次） |
| **更新 Profile** `PATCH /api/profile/update` | 正常使用不會秒改幾十次 | 同一使用者每分鐘 最多幾次 |
| **刪除帳號申請** `POST /api/account/delete-request` | 避免重複送、騷擾或誤觸 | 同一使用者每天 1～2 次即可 |

其餘讀取類（如 `GET /api/profile/me`、探索頁）可視需求再決定是否限流；若流量不大，可先不做。

### 實作方式概覽

1. **依 IP 辨識**  
   從 request 取得 `X-Forwarded-For` 或 `X-Real-IP`（若前面有 reverse proxy／Zeabur），沒有再用 `request.ip` 或類似欄位。同一 IP 在「某個時間窗內」的請求次數累加，超過就 429。

2. **時間窗與計數**  
   - **固定窗**：例如「每 1 分鐘最多 10 次」＝每分鐘 0 秒重置計數。  
   - **滑動窗**：例如「任意連續 1 分鐘內最多 10 次」＝用每個請求的時間戳記判斷，較公平但實作稍複雜。  
   初版用固定窗即可（例如「每分鐘 N 次」）。

3. **存哪裡**  
   - **記憶體（Map / 物件）**：在 Next.js API 裡用一個變數存「IP → 次數 + 上次重置時間」。優點是簡單、無額外服務；缺點是重啟就清空，且若開多個 instance 各算各的，無法跨機器共用。適合單機、流量不大的情況。  
   - **Redis（或 Upstash Redis）**：把「key = IP 或 userId，value = 計數，TTL = 時間窗」存到 Redis。優點是重啟不清空、多 instance 可共用、可做較細的規則；缺點是要多一個服務。Upstash 有免費額度，適合正式環境或之後流量變大。

4. **回傳**  
   超過限制時回傳 **HTTP 429**，body 可為 `{ "message": "請求過於頻繁，請稍後再試" }`，並可加上 `Retry-After` header 告訴客戶端多久後可再試。

### 小結

- **Rate limiting** = 對「誰、在多久內、能打幾次」設上限，超過就拒絕，用來保護伺服器與避免濫用。  
- YigongHub 已對 **登入、建立/更新 Profile、上傳大頭照、刪除帳號申請** 做限制（依 IP 或 userId、固定窗、in-memory）；實作見 `src/lib/rate-limit.ts`、`src/lib/get-client-ip.ts`、`src/middleware.ts` 及各 API route。之後若多實例或流量變大，可改為 Redis／Upstash。

## 四、小結

- **維運面**：建議優先加上 **Rate limiting**，再視需求做 handle 變更與搜尋優化。
- **產品面**：可依使用者回饋決定是否做通知、資料匯出與管理後台統計。
- **體驗面**：SEO、表單即時驗證、錯誤與無障礙可逐步補齊。

以上為靜態分析與建議，實際優先順序可依人力與業務需求調整。
